## 二、每一步做了什么、为什么

### 步骤 1：创建项目目录结构

**执行的操作：**

```powershell
New-Item -ItemType Directory -Path "E:\学习相关\实战项目\笔记RAG" -Force
New-Item -ItemType Directory -Path "E:\学习相关\实战项目\笔记RAG\rag_db" -Force
```

**为什么这样设计：**

```
笔记RAG/              ← 项目根目录，和你的其他实战项目并列
├── rag_db/           ← ChromaDB 的持久化数据目录，放项目内部便于管理
├── venv/             ← 虚拟环境，不提交到版本控制
├── *.py              ← 源代码，直接在根目录，项目小不需要 src/ 子包
```

`rag_db/` 放在项目内部而非系统临时目录，因为这是**持久化数据**——关电脑后索引还在，下次启动直接读，不用重新跑全量索引。`-Force` 参数避免目录已存在时报错中断。

### 步骤 2：编写 `requirements.txt`

**写入的内容：**

```
mcp>=1.0.0
chromadb>=0.5.0
sentence-transformers>=3.0.0
langchain-text-splitters>=0.3.0
```

**为什么是这 4 个库：**

| 库 | 作用 | 为什么选它 |
|----|------|-----------|
| `mcp` | MCP 协议框架，内置 `FastMCP` | 用装饰器定义 Tool，三行代码暴露一个接口给 CodeBuddy；支持 stdio 传输，无需 HTTP 端口 |
| `chromadb` | 向量数据库 | 纯 Python 实现，零配置，数据存本地文件。29 个笔记的向量数据量极小，ChromaDB 完全够用，不需要 Milvus/Qdrant |
| `sentence-transformers` | 加载 BGE-M3 做 Embedding | 你的笔记中英混写，BGE-M3 是多语言模型，中英文混合查询效果好。1024 维向量，精度和速度平衡 |
| `langchain-text-splitters` | Markdown 按标题切分 | 你的笔记是高度结构化的 md（## / ### 标题层级清晰），按标题切分比按字符切更准——每个切片是一个完整知识点，不会从中间截断 |

**没有选用的常见替代方案及原因：**

- `FAISS` 替代 ChromaDB：FAISS 是纯索引引擎，不自带 metadata 管理，ChromaDB 带元数据存储，检索时直接返回源文件路径
- `llama-index` 替代 langchain-text-splitters：过度工程化，你的场景只需切分器，不需要整个编排框架
- `openai` 做 Embedding：依赖网络和 API Key，本地 BGE-M3 完全离线，免费且速度更快

### 步骤 3：编写 `config.py`

**写入的内容：**

```python
NOTES_DIRS = [
    r"E:\学习相关\学习笔记\python基础\py基础-md",
    r"E:\学习相关\学习笔记\前后端-md",
]
CHROMA_DB_PATH = r"E:\学习相关\实战项目\笔记RAG\rag_db"
EMBEDDING_MODEL = "BAAI/bge-m3"
DEFAULT_TOP_K = 5
CHUNK_SIZE = 500
CHUNK_OVERLAP = 80
```

**为什么单独抽一个 config.py：**

1. **路径不硬编码**：如果以后笔记目录变了，只改这一处，三个模块自动生效
2. **参数可调**：`TOP_K`、`CHUNK_SIZE` 这些参数后续可能需要根据检索效果微调，集中管理比散落在各文件里改方便
3. **不写环境变量**：你的场景是单机本地使用，环境变量反而增加配置复杂度

**各参数取值的原因：**

| 参数 | 值 | 原因 |
|------|-----|------|
| `CHUNK_SIZE` | 500 | 你的笔记每个 `###` 小节通常 200-500 字符，500 能完整包裹一个知识点 |
| `CHUNK_OVERLAP` | 80 | 保证跨切片边界的内容不丢失（比如一个 `###` 下一行正好是代码块开头，重叠能兜底） |
| `DEFAULT_TOP_K` | 5 | 你的笔记总量不大（29 个文件），5 个切片基本覆盖所有候选；多了会引入噪声 |
| 笔记路径 | 两个目录分别写 | 源文件不动，不在项目中复制一份，避免数据冗余和同步问题 |

### 步骤 4：编写三个模块的接口注释

**写入的文件：**

- `rag_server.py`：MCP Server 入口，暴露 4 个 Tool
- `indexer.py`：索引模块，负责切分 + Embedding + 入库
- `retriever.py`：检索模块，负责加载模型 + 语义检索

**为什么先写接口注释再写实现代码：**

1. **定分工，避免耦合**：`indexer` 只管"写"，`retriever` 只管"读"，`rag_server` 只管"暴露 Tool"。三个模块通过 `config.py` 和 ChromaDB 间接通信，互不 import 对方的内部函数
2. **接口即契约**：注释里的函数签名明确了输入输出，后续写代码时不会临时改设计
3. **你还没确认**：你是想先看架构再决定，所以我只写接口注释，等你确认后再填实现

### 步骤 5：创建 Python 虚拟环境

**执行的操作：**

```powershell
cd "E:\学习相关\实战项目\笔记RAG"
python -m venv venv
```

**为什么用 venv 而不是全局安装：**

1. **依赖隔离**：`mcp`、`chromadb` 等库的版本不会和你其他项目冲突
2. **可复现**：`requirements.txt` + `venv` 是 Python 项目的标准搭配，以后重装系统或换电脑可以直接重建环境
3. **CodeBuddy MCP 兼容**：MCP Server 的 `command` 指定 `python` 时，需要在正确环境中；用 venv 的绝对路径 `venv\Scripts\python.exe` 最可靠

### 步骤 6：安装依赖并验证

**执行的操作：**

```powershell
.\venv\Scripts\python.exe -m pip install --quiet mcp chromadb sentence-transformers langchain-text-splitters
.\venv\Scripts\python.exe -c "import mcp; import chromadb; import sentence_transformers; import langchain_text_splitters; print('OK')"
```

**为什么用 venv 里的 pip 而不是全局 pip：**

- 直接 `pip install` 可能装到系统 Python 的 site-packages，和你的其他项目混在一起
- 用 `venv\Scripts\python.exe -m pip` 确保装到当前项目的 venv 里

**验证的必要性：**

- `sentence-transformers` 首次安装时会下载 PyTorch 等底层依赖，如果系统缺少 Visual C++ 运行时或 CUDA 库会有隐式错误。import 通过说明环境完全就绪
- 实操中这一步没有问题，4 个库全部通过
